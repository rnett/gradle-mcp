## Executive Summary

The intent of this change is sound and well-motivated. However, the implementation has a critical correctness gap on Windows (the production target): `Path.normalize()` does not resolve filesystem junctions, so the disclosed path will
silently produce empty `rg`/`ast-grep` results. Additionally, the change directly contradicts an existing skill directive, creates an asymmetric response contract (path present on success, absent on all error paths), introduces a DRY
violation split across two call sites, and removes a roadmap item before its full surface area (skill update, error paths, tests) is complete. There are also moderate concerns about the abstraction boundary being crossed and a missing
spec/ADR for the path exposure contract. The change should not be committed as-is.

## Critical (Must Address)

### 1. [Risk] `verifyToolsList` check gate may fail if docs were hand-edited

The `verifyToolsList` task is wired into `check` via `build.gradle.kts`. It regenerates tool descriptions from source and compares them byte-for-byte against the committed docs. The `docs/tools/PROJECT_DEPENDENCY_SOURCE_TOOLS.md` in this
diff appears to have been edited manually rather than regenerated by `./gradlew :updateToolsList`. If the output does not match exactly (whitespace, ordering, schema block wrappers), `check` will throw and the build will fail.

- **Recommendation:** Run `./gradlew :updateToolsList` and commit the regenerated file. Do not manually edit `docs/tools/` files.

---

### 2. [Logic/Arch] `normalize()` does not resolve Windows junctions â€” `rg`/`ast-grep` won't traverse the disclosed path

`sources.sources.normalize()` performs only *lexical* normalization. On Windows, the `SessionViewSourcesDir.sources` directory contains junction reparse points (created by `FileUtils.createSymbolicLink` in `SourceStorageService.kt`). `rg`
does **not** follow directory junctions by default; without `--follow`, it will silently return zero results when it descends into a junction child. The tool description says "enabling direct use with `rg`, `ast-grep`, or other shell tools"
with no mention of `--follow`, making the advertised behavior incorrect on the primary target OS.

The correct call for a usable shell path is `Path.toRealPath()` (which resolves junctions/symlinks to their real targets), or alternatively the CAS path should be surfaced directly.

- **Recommendation:** Replace `sources.sources.normalize()` with `sources.sources.toRealPath()` at both call sites. Update tool description to document any OS/link caveats, or expose the stable CAS path via a new `SourcesDir` interface
  method.

## Major (Highly Recommended)

### 3. [Risk] Direct contradiction between `searching_dependency_sources` skill and the new tool description

`skills/searching_dependency_sources/SKILL.md` line 25 states a hard NEVER directive: "**NEVER** use generic shell tools like `grep` or `find` on the local directory to find dependency sources; they reside in remote caches managed by
Gradle." The new tool description explicitly invites `rg` and `ast-grep` usage on the returned path. An agent using both the skill and this tool will receive directly contradictory instructions. Per AGENTS.md: "Behavior and features MUST be
documented in tool descriptions AND skills."

- **Recommendation:** Update the skill to distinguish "don't use shell tools to *find/locate* sources" from "you *may* use shell tools to *operate on* a path explicitly returned by the MCP tool."

### 4. [Risk] Zero test coverage for `read_dependency_sources` and `search_dependency_sources`

`formatRefreshMessage`, path normalization, or the string assembly will silently break this contract.

- **Recommendation:** Add integration tests asserting the `Sources root:` header appears on the success path and the path value is a real, traversable directory.

### 5. [Doc] `query` field `@Description` contradicts tool-level description for `DECLARATION` mode

### 6. [Arch] `sources.normalize()` leaks ephemeral CAS session-view paths across the tool/service abstraction boundary

The `SourcesDir` abstraction deliberately hides whether backing storage is a CAS entry, a session view with junctions, a merged directory, or a single-dependency layout. `SessionViewSourcesDir` is explicitly documented as ephemeral (scoped
to a single resolution session). By emitting `sources.sources.normalize()` in tool responses, the tool layer leaks internal storage paths to external callers. A caller that stores the path and reuses it across calls may silently operate on
a stale or nonexistent directory. The interface also exposes a distinct `rootForSearch` property, suggesting a design intent to differentiate browsable root from internal path.

- **Recommendation:** Consider introducing a `stableDisplayPath(): Path` method on `SourcesDir` that returns a stable, real, junction-resolved path appropriate for exposure to callers. This keeps the implementation detail inside the service
  layer.

---

### 7. [Arch] No spec/ADR defines the path exposure contract; skills documentation not updated

There is no OpenSpec proposal or design doc in `openspec/changes/` or `openspec/docs/` defining the stability guarantee, type guarantee, or scope guarantee of the disclosed path. The AGENTS.md invariant "Behavior and features MUST be
documented in tool descriptions AND skills" is violated â€” no skill was updated to document the new `Sources root:` response format.

- **Recommendation:** At minimum update the relevant skill(s) to document the new response format. Ideally create a brief design note covering: path stability (is it valid across calls?), what "merged sources root" means for
  single-dependency vs. multi-dependency queries, and OS-specific usage notes.

## Moderate (Recommended)

### 9. [Logic/Arch/Risk] `refreshMessage` (including `Sources root`) is discarded on the error path in `searchDependencySources`

`refreshMessage` is computed unconditionally at line 215, but the error return at line 218 (`return@tool response.error`) discards it. The tool description claims "Returns the absolute path" unconditionally, but this is only true on the
success path. An agent retrying a failed search query loses the path in the error response, despite sources having been successfully resolved.

- **Recommendation:** Prepend `refreshMessage` to all non-`null` return values, including error returns, e.g., `return@tool "$refreshMessage\n\n${response.error}"`.

---

### 10. [Risk/Doc] Three error branches in `readDependencySources` also omit the sources root

Lines 116 (`"Invalid path: ..."`), 129 (`"Path not found: ..."`), and 137 (`"Path is neither a file nor a directory: ..."`) all return after `refreshMessage` (including `Sources root:`) has been computed but before it is included. The tool
description's unconditional claim is violated on all three branches.

- **Recommendation:** Prepend `refreshMessage` (or at minimum the `Sources root:` line) to each of these error returns.

---

### 11. [Clean/Idiom/Arch] DRY violation: duplicated `Sources root` message construction at two call sites

Lines 110 and 215 independently assemble the same two-part composite string:

```kotlin
"Sources root: <normalized-path>\n${formatRefreshMessage(sources.lastRefresh())}"
```

This splits formatting responsibility across two call sites. Any future addition to the response header (e.g., a third metadata field) must be applied twice. `formatRefreshMessage` was already the extracted helper for the refresh portion;
the "Sources root" prefix should be folded in.

- **Recommendation:** Extend or replace `formatRefreshMessage` with a `formatSourcesHeader(sources: SourcesDir): String` private helper that encapsulates the full two-line header construction.

---

### 12. [Logic] `MergedSourcesDir.lastRefresh()` always returns `null`, producing a misleading "not refreshed yet" message for Gradle sources

`GradleSourceService` returns a `MergedSourcesDir` when `gradleSource = true`. `MergedSourcesDir.lastRefresh()` is hardcoded to return `null`. After this change, every Gradle-source response will display:

```
Sources root: C:\...\gradle-sources\...
Sources have not been refreshed yet.
```

even immediately after a successful source extraction. The path is valid and the sources are present, but the message contradicts that.

- **Recommendation:** Implement `MergedSourcesDir.lastRefresh()` to return the most recent refresh time of its constituent parts, or suppress the refresh line when it is known to be unsupported.

`Sources root` + refresh composite.

- **Recommendation:** Rename to `sourcesHeader` or `sourcesMetaHeader`.
